\section{Results}
\label{sec:results}

\subsection{Compression Ratio Evolution}
The repository history shows a clear progression from a simple order-0 baseline toward stronger BWT-based statistical compressors. The key milestones for the purposes of this report are \texttt{v5}, which consolidated the best balanced requirement-compliant implementation; \texttt{v8}, which optimized for speed rather than ratio; and \texttt{v9}, which pushed the statistical line further and became the strongest project variant in compression ratio.

Table~\ref{tab:compression_comparison} shows three distinct optima. \texttt{v9} is the strongest project version in compression ratio at 54.32\%, improving on \texttt{v5}'s 54.77\% and slightly beating bzip2's 54.88\%, although it still remains behind xz at 54.16\%. \texttt{v8} is the clear speed winner among the project versions. Even so, \texttt{v5} remains the most balanced overall design: its ratio stays close to \texttt{v9}, while its compression time is far lower and its output quality is vastly better than \texttt{v8}.

The file-level results show a more nuanced picture. \texttt{v9} wins files A, E, and G; \texttt{v5} wins B and D; xz wins C and H; and bzip2 wins F. Against bzip2 specifically, \texttt{v9} produces smaller outputs on A, B, C, D, E, and G, but loses on F and H. The gains are most convincing on structured inputs such as A and G, where the broader per-block search of \texttt{v9} can choose a better transform/model combination than the fixed \texttt{v5} pipeline.

The contrast with \texttt{v8} is even sharper. On file B, the ratio is 17.35\% for both \texttt{v5} and \texttt{v9}, but 37.84\% for \texttt{v8}; on file G, the values are 28.82\%, 28.48\%, and 72.76\%; and on file H they are 43.79\%, 43.64\%, and 58.02\%. In other words, \texttt{v8} demonstrates that much of the runtime cost in the statistical variants comes from modeling and coding sophistication rather than from file I/O or framework overhead.

From an information-theoretic viewpoint, file D behaves essentially as an incompressible source: all tools remain at about 8 bits/symbol, so the main objective there is minimal overhead rather than real compression. Files F and H instead expose model mismatch. F is only weakly compressible and fairly binary-heavy, so the current statistical line captures less exploitable structure than bzip2, which explains why bzip2 reaches 81.06\% while \texttt{v5} and \texttt{v9} remain at 82.42\% and 81.70\%. H is clearly compressible, but xz reduces it to 2.87 bits/symbol whereas the project variants stay near 3.5 bits/symbol, indicating that substantial longer-range structure remains uncaptured by the present models.

\subsection{Performance Analysis}
From a systems perspective, the repository exposes three practical conclusions. First, stronger modeling is worthwhile on structured inputs because the ratio improvements are real, but the extra search cost can become large. Second, parallel block processing is important for making a BWT-based pipeline practical. Third, once entropy coding and preprocessing are removed, throughput increases dramatically, but the output size deteriorates so much that the result is better seen as a different product goal rather than a direct replacement. This is why \texttt{v5} still stands out as the best overall balance, even though \texttt{v9} compresses slightly better and \texttt{v8} runs much faster.

\subsection{Failed Experiment: Context Mixing}
Between \texttt{v5} and \texttt{v7}, the project explored \texttt{v6}, which attempted PAQ-inspired multi-order PPM with context mixing. The approach used static weight training, train on the first 100KB, freeze weights, and store them in the header.

Results were decisive: \texttt{v6} achieved 59.85\% average ratio (4.93pp worse than \texttt{v5}'s 54.73\%) while running 157$\times$ slower. Key findings: (1) BWT preprocessing \textit{helped} multi-order PPM by 8pp; (2) static weights cannot adapt to different file sections, a fundamental limitation versus PAQ's adaptive mixing; (3) byte-level range coding leaves information on the table compared to PAQ's bit-level arithmetic coding.

This confirmed that successful context mixing requires adaptive weight updates and bit-level coding precision, explaining PAQ's extreme ratios at extreme runtime costs. The lesson influenced subsequent designs: \texttt{v7}/\texttt{v8} prioritized speed, while \texttt{v9} returned to the proven \texttt{v5} statistical core but added per-block transform search instead of model-order complexity.
